# Inline Feedback Output Consistency

**Status:** Shipped — unreviewed (PR [#48](https://github.com/andrewhml/planpong/pull/48), commit `98d27e0`)

## Context

Inline planner mode makes the active agent responsible for showing reviewer feedback, editing the plan, and recording one planner decision per issue. That is the right interaction model, but the user-facing output is currently too dependent on whatever the active agent decides to write.

Observed output:

```text
Round 1 (Direction): 4 P2 issues, 0 P1. Needs revision.

Addressing all four issues now. Let me update the plan.
```

This is concise, but it omits the most important review-loop artifact: a clear table of reviewer issues and the planner's decision for each issue. The user should not have to infer which issues were accepted, rejected, deferred, or silently skipped. Inline mode especially needs this because the active agent is the planner and may revise the plan directly before calling `planpong_record_revision`.

There is also an existing consistency gap between CLI and MCP status/finalization. CLI loop code builds status lines with `phaseExtras` from `runReviewRound`, while MCP calls `writeStatusLineToPlan`, which recomputes stats and drops direction/risk phase extras. That means direction confidence, risk level, blocked state, and related status details can differ depending on entry point.

This plan addresses both problems:

- Add canonical markdown display blocks for round feedback and planner decisions.
- Make inline mode show a table before and after revision.
- Use one shared review-round finalization/status helper for CLI and MCP so status lines and phase-specific fields are consistent.

## Target Output

After `planpong_get_feedback`, the user should see a consistent round summary plus an issue table:

```markdown
Round 1 - Direction - Needs revision

| ID | Sev | Section | Reviewer issue | Planner decision |
|---|---:|---|---|---|
| F1 | P2 | Context | Scope boundary is unclear | Pending |
| F2 | P2 | Steps | Verification criteria are missing | Pending |
| F3 | P2 | Risks | Dependency assumption is untested | Pending |
| F4 | P2 | Rollout | Rollback plan is incomplete | Pending |
```

After inline `planpong_record_revision`, the user should see the same issue set with decisions:

```markdown
Round 1 - Planner decisions

| ID | Sev | Reviewer issue | Decision | Rationale |
|---|---:|---|---|---|
| F1 | P2 | Scope boundary is unclear | Accepted | Added explicit out-of-scope boundaries |
| F2 | P2 | Verification criteria are missing | Accepted | Added test and manual verification steps |
| F3 | P2 | Dependency assumption is untested | Deferred | Needs user confirmation on provider constraints |
| F4 | P2 | Rollback plan is incomplete | Accepted | Added rollback criteria |
```

The exact titles and rationales come from the persisted feedback and response artifacts. The active agent can add short prose around the table, but the table itself should be generated by planpong and presented unchanged.

## Steps

- [ ] Add shared presentation helpers
  - Create `src/core/output.ts` or `src/core/presentation.ts`.
  - Add `formatFeedbackDisplay(args)` returning a markdown string for one review round.
  - Add `formatDecisionDisplay(args)` returning a markdown string after a revision is recorded.
  - Keep these helpers pure: no filesystem reads, no console output, no chalk.
  - Escape pipe characters and normalize newlines so issue titles/rationales cannot break markdown tables.
  - Truncate long issue titles and rationales conservatively, e.g. 140 characters, so the table remains readable.

- [ ] Define a stable issue/decision row model
  - Add an internal type such as:
    - `issue_id`
    - `severity`
    - `section`
    - `title`
    - `decision: "pending" | "accepted" | "rejected" | "deferred"`
    - `rationale?: string`
    - `verified?: boolean`
  - Build rows by joining `feedback.issues[]` to `revision.responses[]` by `issue_id`.
  - If a response exists for an unknown issue ID, include it in an "Unmatched responses" warning field in JSON, but do not put it in the main table.
  - If an issue has no response after revision, show `Missing` in the decision column and return a warning. This should normally be impossible because `record_revision` validates coverage.

- [ ] Update `planpong_get_feedback` response
  - Keep existing content blocks for backward compatibility:
    1. status line as text
    2. JSON payload
  - Add a markdown display block, preferably before JSON or inside JSON as `display_markdown`.
  - JSON payload should include:
    - `display_markdown`
    - `issue_rows`
    - `phase_label`
    - existing `phase`, `verdict`, `issues`, `severity_counts`, `status_line`
  - For feedback rows, every planner decision is `pending`.
  - The display markdown should include direction/risk phase extras when present:
    - Direction: confidence
    - Risk: risk level, risk count, promoted risks
    - Blocked: blocking state

- [ ] Update `planpong_record_revision` response for inline mode
  - Add `display_markdown` containing the issue/decision table.
  - Add `decision_rows` to JSON.
  - Preserve existing fields:
    - `round`
    - `responses`
    - `accepted`
    - `rejected`
    - `deferred`
    - `unverified_rejected`
    - `plan_updated`
    - `status_line`
    - `planner_mode`
    - `idempotent_replay`
  - Include severity and issue title by reading current round feedback and joining by issue ID.
  - If accepted issues exist but `plan_updated` is false, include a visible warning line below the table in `display_markdown`, not only stderr.

- [ ] Update `planpong_revise` response for external mode
  - Add the same `display_markdown` and `decision_rows` fields as inline mode.
  - This keeps external and inline revision output consistent.
  - Include edit telemetry below the table when `revision_mode: "edits"` is active:
    - attempted
    - applied
    - failed
    - recovered
  - Do not require callers to inspect provider-specific logs to understand what happened.

- [ ] Update CLI UI to use the same presentation helpers
  - `printFeedbackSummary` should either call `formatFeedbackDisplay` and print it with minimal color, or use the same row-building helper and render a CLI-friendly table.
  - `printRevisionSummary` should use the same issue/decision rows instead of only printing responses.
  - If full markdown tables are awkward in colored CLI output, keep a CLI-specific renderer but share the row model and table content decisions.

- [ ] Add a shared `finalizeReviewRound` helper
  - Add a helper in `src/core/operations.ts` that handles the common post-review work for CLI and MCP:
    1. Persist `round-N-feedback.json`.
    2. Compute severity trajectory.
    3. Preserve and pass through `phaseExtras`.
    4. Update status line consistently.
    5. Update session status for blocked/approved states.
    6. Update `session.planHash`.
    7. Return canonical display data for the round.
  - CLI and MCP should both call this helper instead of separately building status/finalization behavior.
  - `writeStatusLineToPlan` can remain a lower-level helper, but it should not be the MCP-only path that drops phase extras.

- [ ] Make status-line phase extras consistent
  - Ensure `buildStatusLine` always receives the same phase extras in CLI and MCP:
    - `confidence` for direction
    - `risk_level`, `risk_count`, `risks_promoted` for risk
    - `is_blocked` for blocked
  - Store enough phase-extra metadata in feedback artifacts or recompute it from the feedback object when replaying.
  - Add tests that compare CLI and MCP status line strings for the same mock feedback.

- [ ] Update MCP server instructions and slash prompts
  - In `src/mcp/server.ts`, instruct the active agent to display `display_markdown` exactly after feedback and after revision.
  - Replace loose guidance like "Show the user the feedback summary and issues" with:
    - "Show the `display_markdown` block from the tool response."
    - "Do not replace the issue table with a prose-only summary."
  - Update MCP prompt text for `/planpong:review` and `/planpong:review_interactive` to require the table after every round.
  - Inline mode instructions should say: after `planpong_get_feedback`, show the pending issue table; after editing and `planpong_record_revision`, show the decision table.

- [ ] Add tests for presentation helpers
  - Feedback table with pending decisions.
  - Decision table with accepted/rejected/deferred decisions.
  - Pipe escaping in issue title and rationale.
  - Long rationale truncation.
  - Missing response after revision produces `Missing` row and warning.
  - Unknown response ID produces warning and is not merged into the main table.
  - Verified/unverified issue marker appears when present, if included in table design.

- [ ] Add MCP handler tests

  - `getFeedbackHandler` display table includes every issue exactly once.

  - Given the same risk feedback, CLI and MCP paths produce the same status line including risk level and risk count.
  - Given blocked feedback, CLI and MCP both set session status and display blocked state consistently.
  - Replayed feedback from persisted artifacts can regenerate the same display markdown and phase extras.

## Verification Criteria

- Every non-converged `planpong_get_feedback` response includes a markdown table with one row per reviewer issue and `Planner decision` set to `Pending`.
- Every `planpong_record_revision` response includes a markdown table with one row per reviewer issue and the recorded planner decision.
- Every `planpong_revise` response includes the same decision table shape as `planpong_record_revision`.
- MCP server instructions require the agent to display `display_markdown` rather than replacing it with prose-only summaries.
- Direction status lines include confidence consistently in CLI and MCP.
- Risk status lines include risk level/count consistently in CLI and MCP.
- Blocked status is handled consistently in CLI and MCP.
- Table formatting is stable when issue titles/rationales contain pipes, newlines, or long text.
- `npm test` passes.
- `npm run typecheck` passes.

## Key Decisions

- **Planpong generates the table.** Do not rely on the active agent to invent a table from raw JSON. The tool returns a canonical markdown block.
- **Feedback table first, decision table second.** After feedback, decisions are pending. After revision, decisions are known. Both moments need visible structure.
- **One row model across CLI, MCP, inline, and external.** Different output surfaces can render differently, but they should agree on row content.
- **Shared review finalization over patching MCP only.** The status-line inconsistency is a shared-core issue. Fixing only server instructions would leave CLI/MCP drift in place.
- **Keep JSON fields alongside markdown.** Markdown is for display; structured `issue_rows` and `decision_rows` keep downstream automation possible.

## Risks & Mitigations

| ID | Risk | Mitigation |
|---|---|---|
| R1 | Tool responses become noisy with extra markdown blocks | Keep existing status and JSON blocks; put markdown in JSON and optionally a separate display block. Tests define block order |
| R2 | Agent ignores the generated table anyway | Update MCP server instructions and slash prompts to explicitly show `display_markdown` exactly |
| R3 | Tables get too wide for long issues | Truncate long fields and keep full details in JSON |
| R4 | CLI color formatting diverges from markdown output | Share row-building logic; allow renderer differences only after content is canonical |
| R5 | Shared finalization refactor changes CLI behavior | Add tests with mock feedback comparing old expected status-line semantics, especially direction/risk extras |
| R6 | Replayed/idempotent rounds regenerate slightly different display text | Derive display markdown from persisted artifacts and deterministic formatting, not current provider output |

## Out of Scope

- Changing reviewer prompts or planner prompts.
- Adding a rich terminal table dependency.
- Forcing the active agent to ask user approval before each issue decision.
- Building a full web report UI.
- Solving MCP round idempotency; that is covered by `docs/plans/mcp-round-idempotency.md`.

## Limitations & Future Work

- **The active agent can still add prose around the table.** The tool can provide canonical display markdown, but it cannot fully control the assistant's surrounding text.
- **Decision rationale quality still depends on planner behavior.** The table makes decisions visible; it does not guarantee that accepted/rejected/deferred rationales are good.
- **Very large issue sets may produce long tables.** If reviewers start returning more than ~10 issues per round, add grouping by severity or a collapsed summary, but keep one row per issue in JSON.
- **Shared finalization may overlap with the MCP idempotency plan.** Implement carefully so `finalizeReviewRound` and any future `finalizeFeedback` helper have clear ownership rather than duplicating persistence responsibilities.
